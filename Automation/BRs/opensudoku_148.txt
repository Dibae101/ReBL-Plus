App: OpenSudoku

Bug Report Title: Crash on first cell selection on Android 3.0

Description:
OpenSudoku crashes on Android 3.0 (Honeycomb) when selecting any cell on the game board for the first time. This is an Android 3.0-specific compatibility issue, likely related to hardware acceleration or touch event handling changes. App becomes completely unusable on Android 3.0 devices.

Steps to Reproduce:
1. Launch OpenSudoku app
3. Start a new "easy" difficulty game
4. Select a sudoku puzzle from board list
5. Touch/tap any cell on the game board
6. App crashes immediately

Expected Behavior:
Tapping a cell should select it, show number input pad, and allow input. Game should function normally like on Android 2.x.

Actual Behavior:
App crashes on first cell selection. Possible causes: hardware acceleration incompatibility, touch event handling changes, or view initialization issues in Honeycomb.

Environment:
- Package: org.moire.opensudoku
- Android Version: 3.0 (Honeycomb)

Actual Behavior:
- App crashes immediately when any cell is touched
- Crash occurs specifically on Android 3.0 devices
- No error message shown to user
- App terminates unexpectedly
- Game is completely unplayable on Android 3.0
- Works fine on Android 2.x versions

Crash Details:
- Exception Type: Unknown (no stack trace provided in archived issue)
- Likely exceptions: RuntimeException, NullPointerException, or IllegalStateException
- Symptom: Immediate crash on first cell touch/selection
- Location: Cell selection touch event handler
- Affected Version: All versions before fix
- Affected Platform: Android 3.0 (Honeycomb) only
- Reported Date: March 14, 2011
- Reporter: Swift Panda
- Priority: Medium
- Status: Fixed (March 27, 2011)
- Frequency: 100% on Android 3.0 devices

Technical Analysis:

**Android 3.0 Honeycomb Breaking Changes**:

1. **Hardware Acceleration (Most Likely Cause)**:
Android 3.0 enabled hardware acceleration by default. Custom view drawing code that worked in software rendering often broke with hardware acceleration:

```java
// OpenSudoku likely has custom view drawing
@Override
protected void onDraw(Canvas canvas) {
    // Custom drawing for sudoku grid and cells
    // May use unsupported operations with hardware acceleration
    canvas.drawLine(...);
    canvas.clipPath(...); // Not supported in hardware acceleration
}
```

Fix:
```xml
<!-- AndroidManifest.xml - Disable hardware acceleration for specific activity -->
<activity 
    android:name=".GameActivity"
    android:hardwareAccelerated="false" />
```

Or in code:
```java
// Disable for specific view
sudokuBoardView.setLayerType(View.LAYER_TYPE_SOFTWARE, null);
```

2. **Fragment Lifecycle Changes**:
Android 3.0 introduced fragments. If OpenSudoku was updated to use fragments, lifecycle issues could cause crashes:

```java
// Fragment not properly initialized
@Override
public View onCreateView(LayoutInflater inflater, ViewGroup container, Bundle savedInstanceState) {
    View view = inflater.inflate(R.layout.game_board, container, false);
    // PROBLEM: Trying to access views before fully attached
    sudokuBoard = view.findViewById(R.id.sudoku_board);
    return view;
}
```

3. **Touch Event Propagation Changes**:
Touch event handling changed in Honeycomb:

```java
@Override
public boolean onTouchEvent(MotionEvent event) {
    // On Android 3.0, event coordinates or propagation may differ
    // Need to handle new event types or coordinate systems
    return super.onTouchEvent(event);
}
```

4. **View State Management**:
```java
// View state not properly saved/restored
@Override
protected Parcelable onSaveInstanceState() {
    // Must properly save state for Android 3.0
    return super.onSaveInstanceState();
}
```
